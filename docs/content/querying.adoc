= Content Querying
:toc: right
:imagesdir: images

Content in Enonic CMS is fully indexed and queryable. This page covers how content data is indexed, and how to query it using both string-based and JSON-based query formats.

== How content is indexed

When content is created or updated, all properties are automatically indexed. The CMS uses a schema-driven indexing system where each property is indexed according to its <<#value_types, value type>>.

Content can be queried through the <<../apis/graphql#, GraphQL API>> (Guillotine) or programmatically via the content API. Both APIs accept queries in one of two formats:

* <<#noql, NoQL>> -- a string-based query language
* <<#dsl, Query DSL>> -- a JSON-based query structure

[[value_types]]
== Value types

Every property stored in content has a value type that determines how it is indexed and how it can be queried. The value type is determined by the <<../schemas/form-items#, form item>> used in the content type schema.

[cols="1,1,2"]
|===
|Value Type |Form Items |Index behavior

|String
|TextLine, TextArea, HtmlArea, ComboBox, RadioButton, ContentSelector, MediaSelector, ImageSelector, CustomSelector, Tag, CheckBox
|Indexed as both `text` (analyzed/searchable) and `string` (exact match). For TextLine, TextArea, and HtmlArea, the text value is also added to `_alltext` for fulltext search.

|Long
|Long
|Indexed as both `number` and `string`.

|Double
|Double
|Indexed as both `number` and `string`.

|Boolean
|CheckBox
|Indexed as both `string` and `number` (0/1).

|GeoPoint
|GeoPoint
|Indexed as `string` (`"lat,lon"`) and `number` (for range queries on coordinates).

|LocalDate
|Date
|Indexed as `datetime` and `string`. Format: `yyyy-MM-dd`.

|LocalTime
|Time
|Indexed as `string`. Format: `HH:mm:ss`.

|LocalDateTime
|Instant
|Indexed as `datetime` and `string`. Format: `yyyy-MM-dd'T'HH:mm:ss`.

|DateTime
|DateTime
|Indexed as `datetime` and `string`. Always includes timezone. Format: `yyyy-MM-dd'T'HH:mm:ssZ`.

|Reference
|ContentSelector, MediaSelector, ImageSelector
|Content ID reference. Indexed as `string`.

|BinaryReference
|AttachmentUploader
|Reference to an attachment binary. Indexed as `string`.

|Set
|ItemSet, OptionSet
|Not a value itself -- contains nested properties that are individually indexed using their respective types.
|===


=== _alltext

The system automatically populates a special `_alltext` field by aggregating all text-analyzed properties (TextLine, TextArea, HtmlArea). This field enables broad fulltext search across all textual content without specifying individual field names.


=== Index configuration

By default, content properties are indexed according to their value type. Content types can customize indexing behavior through index configuration to control:

* Which properties are indexed and how
* Whether a property should be included in `_alltext`
* Custom analyzer settings for specific properties

NOTE: For details on index configuration, see the https://developer.enonic.com/docs/xp/stable/storage/indexing[XP indexing documentation].


[[noql]]
== NoQL (string queries)

NoQL is a string-based query language similar to SQL. It supports filtering, ordering, and search functions.

=== Basic syntax

[source]
----
<expression> [ORDER BY <sort>] [LIMIT <count>] [OFFSET <start>]
----

=== Comparison operators

[source]
----
data.rating > 3                               # <1>
data.category = 'news'                        # <2>
data.title LIKE 'enonic*'                     # <3>
data.published != true                        # <4>
data.score >= 4.5 AND data.score <= 9.0       # <5>
----
<1> Greater than (also `>=`, `<`, `\<=`).
<2> Equals.
<3> Wildcard matching with `*`.
<4> Not equals.
<5> Combine with `AND`, `OR`. Use parentheses for grouping.

=== IN operator

[source]
----
type IN ('com.example:article', 'com.example:blogpost')
----

=== Range

[source]
----
data.rating IN [3, 5]       # Inclusive range: 3 <= rating <= 5
data.rating IN (3, 5)       # Exclusive range: 3 < rating < 5
data.rating IN [3, 5)       # Mixed: 3 <= rating < 5
----


=== Query functions

Query functions provide advanced search capabilities within NoQL.

==== fulltext

Searches text-analyzed fields using tokenized matching. The most common way to search content.

[source]
----
fulltext('data.title', 'headless cms', 'AND')              # <1>
fulltext('_alltext', 'enonic content management', 'OR')     # <2>
----
<1> Both words must be present in the title.
<2> Any of the words must be present across all text fields.

Operators: `AND` (all terms), `OR` (any term).

==== stemmed

Similar to `fulltext`, but uses language-aware stemming so that "running" matches "run", "runs", etc.

[source]
----
stemmed('_alltext', 'running quickly', 'AND', 'en')   # <1>
----
<1> The fourth parameter specifies the language for stemming.

==== ngram

Searches using partial word matching (n-gram index). Useful for autocomplete and prefix search.

[source]
----
ngram('_alltext', 'eno', 'AND')    # <1>
----
<1> Matches content containing words starting with "eno" (e.g., "Enonic").

==== range

Explicit range queries, useful for dates and numbers.

[source]
----
range('data.publishDate', instant('2025-01-01T00:00:00Z'), '')     # <1>
range('data.rating', 3.0, 8.0, 'true', 'false')                   # <2>
----
<1> Content published after the given date. Empty string means unbounded.
<2> Fifth and sixth parameters control inclusivity: `'true'` = inclusive, `'false'` = exclusive.

==== pathMatch

Matches content by path hierarchy.

[source]
----
pathMatch('_path', '/my-site/articles/')    # <1>
----
<1> Matches all content under the given path.


=== Date/time functions

When querying date and time fields, use the appropriate function to ensure correct type matching:

[source]
----
data.myDate = dateTime('2025-06-15T12:00:00Z')      # <1>
data.myDate > instant('2025-01-01T00:00:00Z')        # <2>
range('data.created', instant('2025-01-01T00:00:00Z'), instant('2025-12-31T23:59:59Z'))  # <3>
----
<1> `dateTime()` for DateTime values (with timezone).
<2> `instant()` for point-in-time comparisons.
<3> Date range query.

Also available: `date('yyyy-MM-dd')`, `time('HH:mm:ss')`.


=== Sorting

Append `ORDER BY` to control result ordering.

[source]
----
ORDER BY modifiedTime DESC                                 # <1>
ORDER BY data.rating DESC, displayName ASC                 # <2>
ORDER BY geoDistance('data.location', '59.9,10.7') ASC     # <3>
----
<1> Most recently modified first (default for content).
<2> Multiple sort fields.
<3> Sort by distance from a geographic point.

=== Full example

[source]
----
fulltext('_alltext', 'headless cms', 'OR')
  AND type = 'com.example:article'
  AND data.publishDate < instant('2025-07-01T00:00:00Z')
ORDER BY data.publishDate DESC
LIMIT 10
OFFSET 0
----


[[dsl]]
== Query DSL (JSON queries)

The Query DSL is a JSON-based query format that provides the same capabilities as NoQL in a structured format. It is especially useful when building queries programmatically.

=== Basic structure

[source,JSON]
----
{
  "query": { ... },    // <1>
  "sort": { ... }      // <2>
}
----
<1> Query expression (filter, search, boolean compound).
<2> Sort expression.

=== Term expressions

==== term

Exact value match on a single field.

[source,JSON]
----
{
  "term": {
    "field": "type",
    "value": {
      "string": "com.example:article"
    }
  }
}
----

Value types: `string`, `long`, `double`, `boolean`, `instant`, `dateTime`, `localDate`, `localTime`, `localDateTime`.

==== in

Match any of several values.

[source,JSON]
----
{
  "in": {
    "field": "type",
    "values": [
      { "string": "com.example:article" },
      { "string": "com.example:blogpost" }
    ]
  }
}
----

==== like

Wildcard matching.

[source,JSON]
----
{
  "like": {
    "field": "displayName",
    "value": "My*"
  }
}
----

==== range

Range query on a field.

[source,JSON]
----
{
  "range": {
    "field": "data.rating",
    "gt": 3,
    "lte": 8
  }
}
----

Properties: `gt` (greater than), `gte` (greater than or equal), `lt` (less than), `lte` (less than or equal).

For date fields, use typed values:

[source,JSON]
----
{
  "range": {
    "field": "data.publishDate",
    "gte": { "instant": "2025-01-01T00:00:00Z" }
  }
}
----

==== exists

Check that a field has a value.

[source,JSON]
----
{
  "exists": {
    "field": "data.summary"
  }
}
----

==== pathMatch

[source,JSON]
----
{
  "pathMatch": {
    "field": "_path",
    "path": "/my-site/articles/"
  }
}
----


=== Fulltext expressions

==== fulltext

[source,JSON]
----
{
  "fulltext": {
    "fields": ["_alltext"],
    "query": "headless cms",
    "operator": "OR"
  }
}
----

==== stemmed

[source,JSON]
----
{
  "stemmed": {
    "fields": ["_alltext"],
    "query": "running quickly",
    "operator": "AND",
    "language": "en"
  }
}
----

==== ngram

[source,JSON]
----
{
  "ngram": {
    "fields": ["_alltext"],
    "query": "eno",
    "operator": "AND"
  }
}
----


=== Boolean (compound queries)

Combine multiple expressions using boolean logic.

[source,JSON]
----
{
  "boolean": {
    "must": [                                          // <1>
      {
        "term": {
          "field": "type",
          "value": { "string": "com.example:article" }
        }
      },
      {
        "fulltext": {
          "fields": ["_alltext"],
          "query": "headless",
          "operator": "OR"
        }
      }
    ],
    "mustNot": [                                       // <2>
      {
        "term": {
          "field": "data.draft",
          "value": { "boolean": true }
        }
      }
    ],
    "filter": [                                        // <3>
      {
        "exists": {
          "field": "data.publishDate"
        }
      }
    ]
  }
}
----
<1> *must* -- all expressions must match (AND).
<2> *mustNot* -- none of these expressions may match (NOT).
<3> *filter* -- must match, but does not affect relevance scoring.

Also available: `should` -- at least one expression should match (OR).


=== Sorting (DSL)

[source,JSON]
----
{
  "sort": [
    {
      "field": "data.publishDate",
      "direction": "DESC"
    },
    {
      "field": "displayName",
      "direction": "ASC"
    }
  ]
}
----

Geographic distance sort:

[source,JSON]
----
{
  "sort": {
    "field": "data.location",
    "direction": "ASC",
    "type": "geoDistance",
    "value": {
      "lat": 59.9,
      "lon": 10.7
    }
  }
}
----


=== Full DSL example

[source,JSON]
----
{
  "query": {
    "boolean": {
      "must": [
        {
          "term": {
            "field": "type",
            "value": { "string": "com.example:article" }
          }
        },
        {
          "fulltext": {
            "fields": ["_alltext"],
            "query": "headless cms",
            "operator": "OR"
          }
        }
      ],
      "filter": [
        {
          "range": {
            "field": "data.publishDate",
            "lte": { "instant": "2025-07-01T00:00:00Z" }
          }
        }
      ]
    }
  },
  "sort": [
    {
      "field": "data.publishDate",
      "direction": "DESC"
    }
  ]
}
----


== Querying content via GraphQL

The <<../apis/graphql#, Guillotine>> GraphQL API provides a `queryDsl` argument on the content query type, accepting the Query DSL format directly. This is the recommended approach for headless applications.

[source,graphql]
----
{
  guillotine {
    queryDsl(
      query: {
        boolean: {
          must: [
            { term: { field: "type", value: { string: "com.example:article" } } }
            { fulltext: { fields: "_alltext", query: "headless", operator: OR } }
          ]
        }
      }
      sort: {
        field: "data.publishDate"
        direction: DESC
      }
      first: 10
    ) {
      displayName
      ... on com_example_Article {
        data {
          title
          summary
          publishDate
        }
      }
    }
  }
}
----

NOTE: For complete Guillotine query documentation, see the https://developer.enonic.com/docs/guillotine[Guillotine documentation].


== Common query patterns

=== Content by type
[source]
----
type = 'com.example:article'
----

=== Content by path
[source]
----
_path LIKE '/my-site/articles/*'
----

=== Published content only
[source]
----
publish.from < instant('2025-06-15T00:00:00Z')
  AND (publish.to > instant('2025-06-15T00:00:00Z') OR publish.to NOT EXISTS)
----

=== Fulltext search with type filter
[source]
----
fulltext('_alltext', 'search terms', 'OR')
  AND type IN ('com.example:article', 'com.example:page')
----

=== Recently modified content
[source]
----
modifiedTime > instant('2025-06-01T00:00:00Z')
ORDER BY modifiedTime DESC
----

=== Content referencing another item
[source]
----
data.relatedArticle = 'content-id-here'
----

=== Geolocation proximity
[source]
----
ORDER BY geoDistance('data.location', '59.9139,10.7522')
----
